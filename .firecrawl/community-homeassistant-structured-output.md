Something I noticed, is in the account settings on open router under "training, logging, & privacy" there are options to enable endpoints that train on inputs, or publish prompts.

Certain free models are indeed only available through endpoints that are marked as training on user prompts. This is the case of Kimi k2 (free) i wanted to try using.

Unfortunately, in the integration logs this isn't immediately obvious, I suspect the API doesnt specify that this is the reason either.

So for some free models, it'll never work unless you accept models that train on prompts.

It makes sense after all that they are free in exchange for helping train them. Maybe a notice somewhere in the interface could be useful to explain why this happens?

For free models not concerned by this, it is inconsistent how available they are.

What ive personally done is pick a very cheap model and use that, it seems more consistent, and negligble in terms of cost.

My experimenting so far with Gemini 2.5 light was as cheap as $ 0.0000617 for some simple instructions.

For those looking for some reliability, I'd recommend trying that for now. Open router doesn't require payment under $1 in usage, and you can set a limit to the API key as well.

Hope this helps!

I want to summarize what has been observed across the related issues and why this is still failing for many of us even after [#149873](https://github.com/home-assistant/core/pull/149873) was merged.

### Related history

- [OpenRouter integration unexpected error on +Add...either option #149905](https://github.com/home-assistant/core/issues/149905) — “Unexpected error on +Add…” (closed as fixed by [Bump python-open-router to 0.3.1 #149873](https://github.com/home-assistant/core/pull/149873))
- [Bump python-open-router to 0.3.1 #149873](https://github.com/home-assistant/core/pull/149873) — Bumped `python-open-router` to 0.3.1
- [OpenRouter AI Tasks fail with "Error talking to API" #150294](https://github.com/home-assistant/core/issues/150294) — Ongoing “Error talking to API” failures (this issue)

Despite the dependency bump in [#149873](https://github.com/home-assistant/core/pull/149873), we are still seeing multiple classes of failure that point to structural fragility in the integration rather than a single bug.

* * *

# Model parsing is not forward-compatible

Recent logs show:

```
ValueError: 'audio' is not a valid Modality
ValueError: 'embeddings' is not a valid Modality
InvalidFieldValue: Field "output_modalities" ...
```

This indicates:

- The integration (via `python-open-router`) uses a strict `Modality` enum.
- When OpenRouter introduces new modalities (`audio`, `embeddings`, etc.), deserialization fails.
- The failure aborts the config flow entirely.

This is not a version issue: OpenRouter is evolving rapidly, and new modalities will continue to appear. The integration currently assumes a closed modality set and fails hard instead of:

- Ignoring unknown modalities
- Gracefully skipping unsupported models
- Falling back to string-based parsing

This means every upstream capability expansion risks breaking Home Assistant again.

* * *

# Tool injection when model does not support tools

Multiple reports show:

```
404 - No endpoints found that support tool use
```

Findings from testing:

- When “Enable Home Assistant Control” is enabled, the integration sends device tools.
- Some models (especially free-tier ones) do not support tool use.
- OpenRouter correctly rejects the request.
- In some cases, tools appear to be sent even when control is disabled.

The integration should:

- Filter model selection based on `tools` capability
- Or dynamically retry without tools if unsupported
- Or prevent tool injection when HA control is disabled

Right now, users are allowed to select incompatible models and only discover incompatibility at runtime via opaque 404 errors.

* * *

# Structured output schema incompatibilities

Example error:

```
Invalid schema for response_format 'frontend__automation__save':
'additionalProperties' is required to be supplied and to be false.
```

This suggests:

- HA sends structured output schemas.
- Some providers enforce stricter JSON schema validation.
- The integration does not adapt schemas per provider.

Result: AI Tasks fail even when the same model works via the OpenAI integration.

* * *

# Streaming tool argument crash

Observed:

```
TypeError: the JSON object must be str, bytes or bytearray, not NoneType
```

This happens because:

```
json.loads(arguments)
```

is called without checking if `arguments` is `None`.

That is a simple defensive coding fix.

* * *

# Dependency isolation issues

We’ve also seen:

```
ImportError: cannot import name 'omit' from openai._types
```

This indicates dependency/version conflicts with `openai`.

This suggests the integration environment is not fully isolated or properly pinned, which makes it vulnerable to upstream packaging changes.

* * *

# Big Picture

The recurring failures suggest the integration is:

- Strict where it should be tolerant (enum parsing)
- Not filtering models by capability
- Not adapting behavior per provider
- Not defensive against partial/nullable responses
- Highly sensitive to upstream API changes

This explains why it repeatedly breaks when:

- OpenRouter adds new modalities
- Free-tier routing changes
- Providers tighten schema validation
- New model capabilities are introduced

* * *

# Suggested Direction (Architectural)

To stabilize this integration long-term:

1. Make modality parsing forward-compatible (ignore unknown enum values).
2. Filter models by capability (`tools`, `structured_outputs`, multimodal support).
3. Do not send tools when HA control is disabled.
4. Add safeguards around tool argument decoding.
5. Consider provider-aware schema shaping.
6. Ensure proper dependency pinning/isolation.

Without these changes, every upstream OpenRouter change risks another regression.

* * *

This integration is extremely powerful and valuable, but right now it behaves as if upstream APIs are static, and they are not.

Would appreciate clarity on:

- Whether a broader hardening pass is planned
- Or if a contributor PR addressing these points would be welcome

Happy to help test or assist with structured fixes. I'll gladly work on these points so long as the help would be welcomed and desired. [@homeassistant](https://github.com/homeassistant) [@joostlek](https://github.com/joostlek)
